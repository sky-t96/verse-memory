# UEFN / Verse 最上級ルール・マニフェスト（全プロジェクト共通）

> 本ドキュメントは、すべての新規プロジェクトで最初に適用する「最上位ルール」です。  
> 実装・レビュー・AI支援（Cursor / Claude Code）すべての判断基準として扱います。

---

## 0. 目的

- バグを減らし、再現性のある実装を行う
- 仕様変更に強い構造を維持する
- 公式情報を根拠にし、思い込み実装を排除する
- AIの生成コードを「安全に使える状態」にする

---

## 1. 最上級原則（優先順位）

1. **公式一次情報 > 実装経験 > 推測**
2. **動くコードより、壊れにくいコード**
3. **1つの真実源（Single Source of Truth）**
4. **副作用の局所化（no_rollback 呼び出しの集約）**
5. **イベント処理は冪等（同じイベントが複数来ても壊れない）**
6. **運用で守るより、構造で守る**

---

## 2. 実装前チェック（必須）

- [ ] 公式ドキュメントURLを確認したか
- [ ] 失敗コンテキスト / no_rollback の衝突を確認したか
- [ ] イベント重複発火時の挙動を設計したか
- [ ] ラウンド跨ぎデータの保存方針を決めたか
- [ ] UI表示値とゲーム内実値の真実源を統一したか
- [ ] 終了条件（正常系 / 異常系）を分離したか

---

## 3. Verse 言語ルール（厳守）

## 3.1 呼び出し・エフェクト

- `decides` 系APIは **`[]` 呼び出し**で扱う
- `if` 条件には、失敗可能式のみを置く
- `no_rollback` を失敗コンテキストで呼ばない
- `<suspends>` と `<decides>` を同一関数で併用しない
- `spawn` は単一コルーチン呼び出しを原則とする

## 3.2 制御構文

- `break` は `loop` 内のみ使用（`for` 直下で使わない）
- ループ内副作用は最小化し、ハンドラ側に寄せる
- 比較・アンラップ・配列アクセスの失敗可能性を常に考慮する

## 3.3 型とメッセージ

- UI表示は `message` 型で統一する
- 動的文字列は `<localizes>` 関数経由で組み立てる
- `string` 直渡しを避ける（HUD / SetText）

---

## 4. データ永続化ルール

## 4.1 永続方式

- ラウンド/参加跨ぎの永続は `weak_map(player, persistable_class)` を基本にする
- 値型は `persistable class` を採用し、フィールド追加可能な設計にする
- 永続データは集約し、マップ乱立を避ける

## 4.2 競合回避

- `stat_creator_device` 永続と Verse 永続の二重管理をしない
- 保存ソースは1つに固定する
- `Persist` 関数に責務を混在させない（例: 得点保存とラウンド進行更新を分離）

---

## 5. デバイス/API 運用ルール

## 5.1 no_rollback 運用

- no_rollback API呼び出しは集約・最小化する
- `loop` 本体からの直接呼び出しを禁止
- イベントハンドラまたは専用適用関数で処理する

## 5.2 イベント順序・多重発火

- イベント順序は非保証前提で設計
- 同一イベントの重複発火を想定し、冪等ガードを入れる
- 勝敗や終了処理には二重ロック（ラウンド単位 + マッチ単位）を置く

## 5.3 Timer/Respawn/Teleport

- `timer_device` は参加者0人開始ケースに対応する
- タイマー不発環境のフォールバック表示を持つ
- 撃破直後処理は短遅延で安定化する
- テレポート競合時は Transform + `FortCharacter.TeleportTo` を優先検討する

---

## 6. マッチ進行ルール（共通）

- 正常終了と異常終了を分離する
- 中断フロー（退出・タイムアウト）は専用関数で扱う
- ラウンド完了更新は副作用の成功可否に依存させない
- 参加者全員の進行状態を同期更新する
- 新規マッチ判定は明示的な進行カウンタで行う

---

## 7. UI / MVVM ルール

- `stat_creator_device` を中心に UI とゲームロジックを同期
- 表示用値と実計算値の二重管理を禁止
- 表示更新は再生成ではなく差分更新を優先
- 演出（HUD/Audio/VFX）は Direct Event Binding を優先活用

---

## 8. AI利用ルール（Cursor / Claude Code）

- AI出力は「提案」であり「確定仕様」ではない
- すべての重要処理に対して、以下を必須化する  
  - 公式根拠URL  
  - 想定失敗ケース  
  - ロールバック不能操作の位置  
  - 終了条件と中断条件
- AIが曖昧な説明をした場合は採用しない
- 「動く」だけでマージしない（再現性・拡張性を確認）

---

## 9. レビュー基準（PR / 変更承認）

- 正確性: 公式根拠と一致しているか
- 安全性: no_rollback / failure context 衝突がないか
- 一貫性: 既存アーキテクチャと整合するか
- 冪等性: 多重イベントで壊れないか
- 可観測性: 必要なログがあるか
- 後方互換性: 永続データ更新で破壊がないか

---

## 10. 例外ルール

- 例外採用時は、必ず以下を記録する  
  - 例外理由  
  - 影響範囲  
  - 監視方法  
  - 解除条件（いつ標準ルールに戻すか）

---

## 11. 変更管理

- このマニフェストは固定ではなく、公式更新に追従する
- 更新時は以下を必須にする  
  - 変更履歴（日時 / 変更者 / 理由）  
  - 参照URL差分  
  - 既存プロジェクトへの影響評価

# Verse 教科書ルール（公式準拠・プロジェクト適用版）

> 以下は、公式ドキュメントの内容をプロジェクト運用に落とし込んだ「実装ルール集」です。

## A. 言語基礎

- 式と文の境界を意識し、失敗可能式の扱いを明確にする
- 配列アクセス・オプションアンラップ・比較条件を安全に扱う
- 関数のエフェクト（`decides` / `suspends` など）を分離設計する

## B. Failure Context / Specifiers

- 失敗コンテキストで禁止される呼び出しを事前に排除
- 条件節に重い副作用を入れない
- no_rollback の呼び出し位置を固定化する

## C. Concurrency

- `spawn` 乱用禁止、責務単位で非同期関数を切る
- タスクリーク防止（多重起動ガード）を入れる
- ループ駆動は停止条件を必ず持つ

## D. Persistence

- `weak_map + persistable class` で進化可能な保存構造を作る
- 永続スキーマ変更はデフォルト値付き追加を基本にする
- 保存・復元の責務を明示し、暗黙更新を禁止する

## E. Devices

- デバイスごとの初期化タイミング差を吸収する待機を設計
- `timer_device` は参加者状態で起動分岐する
- 競合しやすい処理（Teleport/Respawn/Elimination）は順序非保証で組む

## F. UI

- UI表示値は必ず真実源（Stat等）から導出
- 文字列は `message` と `localizes` を基本とする
- 演出は Direct Event Binding を優先し、Verseはトリガ発火中心にする

## G. テスト

- 1人テスト、同時脱落、途中退出、0参加開始を必ず検証
- ラウンド跨ぎ・再参加・継続マッチの復元を確認
- 失敗時フォールバックの実行ログを確認

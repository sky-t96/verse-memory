# Verse Memory — 日常開発統合タスク

**目的：** 日々のUEFN開発作業の中で、Verse Memoryのデータ蓄積とマーケティング素材の生成を同時に行う  
**共有先：** プロダクト総合開発チーム（Cursor環境）  
**更新日：** 2026年3月30日

-----

## ワークフロー全体像

```
通常のUEFN開発作業
    ↓
AIがVerseで誤った出力をした／制約にハマった
    ↓
┌──────────────────────────────────┐
│  STEP 1：スキーマレコード抽出       │  ← データ蓄積（プロダクト資産）
│  STEP 2：マーケティング素材化       │  ← 投稿素材（認知・ベータ獲得）
└──────────────────────────────────┘
    ↓
開発作業に戻る（中断は最小限）
```

-----

## STEP 1：スキーマレコード抽出（Cursorで実行）

### トリガー条件

以下のいずれかが発生した時点で、このタスクを起動する：

- AIが非推奨・存在しないVerse構文を提案した
- AIの出力がコンパイルエラーになった（Verse固有の原因）
- デバイスの接続方法・プロパティ設定をAIが誤った
- 実機テストで意図しない挙動が発生し、原因がVerse制約だった
- 過去に解決済みの問題をAIが再び間違えた

### スキーマ定義ファイル

レコード生成時は必ず以下のファイルを参照すること：

```
verse-memory/schema/record_schema.json
```

スキーマの詳細（フィールド定義・カテゴリ一覧・severity基準）は上記ファイルに集約されている。Cursorのセッションが変わった場合も、このファイルを読み込ませることでフォーマットの一貫性を保つ。

### 実行プロンプト

Cursorに対して以下を実行する：

```
以下のスキーマ定義ファイルを参照してください：
verse-memory/schema/record_schema.json

このスキーマに完全に準拠した形で、今発生した問題を
JSONレコードとして出力してください。

必須フィールド：
- trigger（keywords + code_patterns の両方）
- problem（common_ai_mistake を必ず含める）
- solution（正しいコード例があれば含める）
- verification（status, verified_by, verified_date, uefn_version）
- severity（critical / warning / info）
```

### 出力先

- `verse-memory/data/draft/` に保存（レビュー前）
- レビュー後 `verse-memory/data/verified/` に移動
- ファイル命名規則：`{連番}_{キーワード}.json`

### 所要時間目安

1件あたり **2〜3分**（Cursorが自動生成→レビュー→保存）

-----

## STEP 2：マーケティング素材化

### STEP 1の完了直後に行う

スキーマレコードの内容を、以下のフォーマットでX投稿用テキストに変換する。

### 投稿テンプレート

```
【AIがVerseでやりがちなミス】

❌ AIの出力：
{AIが生成した誤ったコードまたはアプローチ（1〜3行）}

✅ 正解：
{正しいコードまたはアプローチ（1〜3行）}

💡 なぜ間違うか：
{1文で原因を説明}

#UEFN #Verse #FortniteCreative
```

### 素材の保管

- `verse-memory/marketing/posts/` に下書きとして保存
- ファイル命名規則：`{日付}_{トピック}.md`
- 投稿タイミングは即時でなくてよい（ストックしてまとめて投稿可）

### 補足：スクリーンショットの取得

- STEP 1の作業中にAIの誤出力画面をスクショしておく
- 投稿時にビフォーアフター画像として添付すると効果的
- 必須ではない（テキストだけでも投稿可）

### 所要時間目安

1件あたり **1〜2分**（テンプレートに流し込むだけ）

## 運用ルール

### 頻度

- 発生ベース（無理に件数ノルマを設けない）
- 目安：週2〜3件が自然に出る想定

### 品質基準

- スキーマレコード：実機検証済みのもののみ登録
- 投稿素材：他の開発者が読んで「あるある」と思えるもの

### 投稿の優先度判断

以下に該当するものは投稿優先度が高い：

1. **severity: critical** のもの（コンパイルが通らない系）
2. 多くの開発者がハマりそうな汎用的な制約
3. 公式ドキュメントに記載がない、または見つけにくい情報

### やらないこと（スコープ外）

- 動画制作（テキスト投稿の反応を見てから判断）
- 投稿の反応分析（初期は投稿を出すことだけに集中）
- 価格設計・ビジネスモデルの検討（Phase 1以降）

## マイルストーン

| 指標 | 目標 | 判断タイミング |
|---|---|---|
| スキーマレコード数 | 30件 | Phase 1（API構築）開始の前提条件 |
| X投稿数 | 15件以上 | 投稿パターンの確立確認 |
| ベータ候補の反応 | 5人以上のDM・リプライ | MCPサーバー公開判断の材料 |

-----

## チーム内での運用開始手順

1. 本ドキュメントをチームに共有
2. 上記ディレクトリ構成をリポジトリに作成
3. `record_schema.json` をスキーマ定義ファイルとして配置
4. 次回の開発セッションから、トリガー条件に該当した時点でSTEP 1→2を実行
5. 週次で蓄積状況を確認（レコード数・投稿数のカウントのみ）
